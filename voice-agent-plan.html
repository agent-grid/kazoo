◍ Build Plan · Voice TUI

Talk to a coding agent.
Hear it think out loud.

A terminal app where you hold a voice conversation with a real coding agent. You speak; it works; and it narrates what it's doing — in first person, semantically — instead of reading raw commands aloud.

TypeScript · one runtime OpenAI Realtime Claude Agent SDK Ink TUI voice-memory.md Twilio Python / Pipecat SDK

01 The idea

Two layers, one persona. A powerful executor does the real work; a thin voice narrator is its ears and mouth. The user never feels two systems — the voice speaks as the harness.

Crucially, the narrator is not the brain. It doesn't try to answer coding questions itself. It listens, routes the task to the executor, and voices what the executor reports. All the hard tool-calling complexity stays inside the agent that's optimized for it.

03 The core loop

You speak a task
Realtime captures audio, runs VAD, and emits a final transcript — conversation.item.input_audio_transcription.completed
Route to the executor
Your loop forwards that transcript to the Claude Agent SDK. The narrator does not answer it — no Realtime function-calling needed.
Executor streams events
Claude emits assistant text (intent preambles = narration gold), tool_use (ground truth), and tool_result (outcomes), live.
Translate to narration
Harvest the preambles, summarize tool calls semantically ("opening the auth module", not "cat src/auth.ts"). Flow or high-level mode decides batching.
Inject & speak
Push narration into Realtime — conversation.item.create + response.create — and it's voiced in the agent's own voice. (This primitive already exists as requestWrapUp().)
Barge-in any time
server_vad auto-cancels the in-flight response when you start talking. Interruption is already handled by the lifted client.

04 Design principles

Narrator ≠ brain

Realtime listens, routes, and voices. Claude reasons and codes. Keep the boundary hard or the voice will improvise wrong answers.

Harvest preambles

Don't build a command→English translator. The executor already states intent in plain language before each tool call — that is the narration.

Two modes

Flow — live play-by-play. High-level — waits, then a clean summary. Same event stream, different batching policy.

Never block the voice

Coding is slow and bursty. Narrate as events arrive; fill long tool calls with progress, never freeze the turn on task completion.

06 Memory & auto-improvement

One markdown file the agent keeps

Appended at the end of a call, prepended into the next call's instructions. Two kinds of memory:

  • Project facts & conventions — like a CLAUDE.md the executor reads ("the API lives in src/server").
  • Voice / narration preferences — "be terse", "don't read paths aloud", "default to high-level mode".

SYNERGY → the wrap-up turn already generates a recap on hangup. Extend that same step to distill "what I learned" and append it. ~30 lines.

07 Risks & what to derisk first

#1 RISK

Terminal audio I/O

The lifted Realtime client assumes a renderer captures mic & plays audio. A TUI has none — you need Node-native audio, which is the only native-dependency gotcha in the whole stack.

  • Prove a mic → Realtime → speaker round-trip before anything else. If that works, the rest is glue.
  • Secondary: keep the executor event log structured from day one, so a fidelity eval can bolt on later without a rewrite.
